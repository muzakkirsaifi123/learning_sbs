    Radar-mongoatlas-flow¶

    Explain the complete radar testing flow for the mongo operator — how internal tests work, what namespace they deploy to, how Istio interacts, and why clouddev (obaas-1) fails while escln157 (radar) works. Here's the complete explanation of the radar testing flow for the mongo operator, including the Istio interaction and why the two environments behave differently:

How Istio Interacts This is the critical piece. Istio's service mesh enforces strict mTLS and egress policies: The mongo-operator needs to reach: MongoDB Atlas API (cloud.mongodb.com) — REST API calls for user/role management MongoDB Atlas cluster (.mongodb.net) — direct driver connection to create databases The radar internal test pod needs to reach: MongoDB Atlas cluster — to verify connections with the secrets the operator created Istio controls this via: Egress Gateways — all external traffic from the mesh must go through the tenant's egress gateway ServiceEntry — defines which external hosts are reachable DestinationRule — TLS origination settings Sidecar resources — controls what each namespace can see If the ServiceEntry for *.mongodb.net or cloud.mongodb.com doesn't exist for a given namespace/tenant, the Istio sidecar will block the outbound connection. 1. Why clouddev (obaas-1) Fails While escln157 (radar) Works Here's the likely root cause based on the architecture: Aspect escln157 (namespace: radar) clouddev (namespace: obaas-1) Namespace Dedicated radar namespace Shared tenant namespace obaas-1 Istio egress config Has ServiceEntry + egress rules for MongoDB endpoints configured for testing May be missing or misconfigured MongoDB egress rules Sidecar injection Either no sidecar on the radar job pod, or correct egress policies exist Sidecar injected, but MongoDB endpoints not in the allowed egress list for that tenant Network policies Permissive for testing namespace Stricter policies on a "real" tenant namespace The most probable scenarios: Scenario A — Istio egress not configured for obaas-1: In escln157, the radar namespace has explicit Istio ServiceEntry resources allowing traffic to *.mongodb.net and cloud.mongodb.com In clouddev, the obaas-1 namespace is a standard tenant namespace that does NOT have those ServiceEntries When the radar pod (in obaas-1) tries to connect to Atlas, the Istio sidecar intercepts and drops the connection because there's no matching egress rule Scenario B — Sidecar injection behavior differs: The radar namespace in escln157 might have the label istio-injection: disabled or the radar Job has sidecar.istio.io/inject: "false" annotation The obaas-1 namespace in clouddev has mesh injection enabled, so the radar Job gets a sidecar that blocks unmeshed outbound traffic Scenario C — The operator itself works, but the radar test pod can't verify: The mongo-operator runs in kube-management which likely has its own egress rules The operator successfully creates databases and secrets (it can reach Atlas) But when the radar pod in obaas-1 tries to verify by connecting to Atlas directly, Istio blocks it In escln157, the radar namespace has the right egress rules so the verification succeeds 6. How to Diagnose / Fix To confirm which scenario it is: Check if the radar pod gets a sidecar in obaas-1: kubectl get pod -n obaas-1 -l app=radar -o jsonpath='{.items[0].spec.containers[].name}' If you see istio-proxy alongside the test container, the sidecar is injected. Check ServiceEntry in the namespace: kubectl get serviceentry -n obaas-1 kubectl get serviceentry -n radar # on escln157 for comparison Check if the test fails at connection or at assertion: If it times out → Istio is blocking outbound If it gets an auth error → Network works, credentials issue Fix options: Add a ServiceEntry for MongoDB Atlas endpoints in the obaas-1 namespace Or add sidecar.istio.io/inject: "false" to the radar helm chart's Job pod template Or configure TENANT_NAMESPACE to a namespace that has the correct egress rules (like what escln157 does by using a dedicated radar namespace) The key takeaway: the radar testing framework is designed to run in a namespace with proper Istio egress configuration for MongoDB. The escln157 environment uses a dedicated radar namespace with those rules, while clouddev uses obaas-1 which is a standard tenant namespace that likely lacks the MongoDB egress ServiceEntry configuration. The Complete Radar Testing Flow for Mongo Operator — Deep Dive PART 1: Why a "Setup" Stage Exists The radar pipeline has 4 stages in order: .pre (initialize) → setup → execution → internal-reports → report The setup stage exists because internal tests run inside the cluster as a Kubernetes Job, NOT on the CI runner. Think about it this way: The CI runner (GitLab) can talk to the Kubernetes API, it can apply YAML manifests But internal tests need to verify things like "can I connect to MongoDB Atlas from inside the cluster?" — that means the test code must execute FROM INSIDE a pod So before the internal tests can verify that the mongo operator works, somebody needs to create the test resources (CRs) first. That's the setup stage's job. The setup stage runs on the CI runner (which has kubectl access) and applies Kubernetes manifests to prepare the cluster for the tests that will run inside it later. PART 3: What Happens After the CRs Are Applied (Operator Reconciliation) Now we enter the domain of the mongo-operator itself. This is NOT triggered by the radar pipeline — it's an autonomous Kubernetes process that reacts to CR creation. Where the operator lives: Deployment name: mongoatlas-operator Namespace: kube-management Watches: ALL namespaces (because WATCH_NAMESPACE: "" in its deployment) The operator is ALREADY running before radar even starts. It was deployed during environment setup (by sbcp-env-setup using the mongoatlas-operator Helm chart). The reconciliation loop: The operator's controller-runtime framework has a Watch set up: // From mongoatlas_controller.go err = c.Watch(source.Kind(mgr.GetCache(), &dbv1.MongoAtlas{}, &handler.TypedEnqueueRequestForObject[dbv1.MongoAtlas]{})) This means: "whenever a MongoAtlas CR is created/updated/deleted anywhere in the cluster, add a reconcile request to my queue." What reconciliation does for MongoAtlas CR specifically: func (r *ReconcileMongoAtlas) Reconcile(ctx context.Context, request reconcile.Request) (reconcile.Result, error) { // 1. Fetch the CR from Kubernetes instance := &dbv1.MongoAtlas{} err := r.client.Get(ctx, request.NamespacedName, instance) 

    // 2. If CR is new (status is empty), create the database
isMongoDBAtlasDatabaseToBeCreated := reflect.DeepEqual(instance.Status, dbv1.MongoAtlasStatus{})
if isMongoDBAtlasDatabaseToBeCreated {
    err = createMongoDBAtlasDatabase(reqLogger, instance, r.atlasClient)
    // 3. Update the CR status to "created"
    err = r.client.Status().Update(ctx, instance)
}
// 4. Requeue after reconciliation time (120 seconds)
return reconcile.Result{RequeueAfter: r.reconciliationConfig.Time}, nil
    } createMongoDBAtlasDatabase does: Connects to MongoDB Atlas using the connection string from the operator's secret (ATLAS_CONNECTION_STRING) Lists all databases that match the namespace prefix pattern (^{namespace}_.) If the database already exists → skips creation, but if CREDENTIAL_SCOPE=namespace, updates the tenant user's access list If the database doesn't exist → creates it by inserting a dummy document into a Operator_Init collection If CREDENTIAL_SCOPE=namespace → calls the Atlas REST API to update the tenant user with the new database in their access list Updates the CR status to created Meanwhile, other CRs trigger their own controllers: MongoAtlasUser CR → the mongoatlasuser controller creates an Atlas database user, generates credentials, creates a Kubernetes Secret (test-from-radar-mongodb) in the same namespace MongoAtlasReadOnlyUser CR → the mongoatlasreadonlyuser controller creates a read-only Atlas user, stores credentials in a secret Timeline: T+0s: kubectl apply creates CRs in TENANT_NAMESPACE T+0-2s: Kubernetes API notifies controller-runtime (via Watch) T+2-5s: Reconcile functions execute (create DB, create users) T+5-30s: Atlas API calls complete, Secrets are created in TENANT_NAMESPACE The operator is event-driven, not polled. The creation of CRs triggers an immediate reconcile. After that, it re-reconciles every 120 seconds to maintain desired state. PART 5: The Dependency Chain (Why Order Matters) Setup stage applies CRs │ ▼ (triggers Watch event) Operator reconciles: - Creates Atlas DB - Creates Atlas users - Creates K8s Secrets in TENANT_NAMESPACE │ ▼ (Secrets now exist) Internal test Job deployed into TENANT_NAMESPACE - Reads those Secrets - Connects to Atlas using those credentials - Verifies roles, permissions, connectivity If you skip setup → no CRs exist → operator has nothing to reconcile → no secrets → internal tests fail. If the operator is broken → CRs exist but are never reconciled → no secrets → internal tests fail. If the network is blocked (Istio) → CRs exist, operator reconciles (operator is in kube-management which has Atlas egress), secrets exist, BUT the test pod in TENANT_NAMESPACE can't reach Atlas → connection tests fail. PART 6: Why clouddev (obaas-1) Fails While escln157 (radar) Works Now the critical difference. Both environments run the same pipeline code, same setup, same operator. The difference is the TENANT_NAMESPACE value: Environment TENANT_NAMESPACE What this means escln157 radar A namespace dedicated for testing clouddev obaas-1 A real tenant namespace used for applications The Istio egress problem: The mongo-operator itself lives in kube-management and can always reach Atlas because that namespace has the necessary ServiceEntry/EgressGateway rules. But the radar test pod lives in TENANT_NAMESPACE. When it tries to connect to Atlas (.mongodb.net): The pod's outbound TCP connection is intercepted by the Istio sidecar (injected automatically because the namespace has istio-injection: enabled) The sidecar checks: "is *.mongodb.net an allowed external destination for this namespace?" It looks for a ServiceEntry that matches In escln157 (radar namespace): The radar namespace was specifically created for testing It either has the correct ServiceEntry for MongoDB Atlas, OR it might not have Istio injection at all (testing namespaces often opt out) The test pod can reach Atlas → tests pass In clouddev (obaas-1 namespace): obaas-1 is a standard tenant namespace with full Istio mesh participation The Istio egress rules for this namespace are configured for the applications that run there (APIs, services) MongoDB Atlas (.mongodb.net, cloud.mongodb.com) may NOT be in the allowed egress destinations for obaas-1 The radar test pod's connection to Atlas gets blocked by the sidecar → connection timeout → test fails But wait — the operator works fine in both environments. That confirms it's not an Atlas API issue. The operator (in kube-management) can create databases and secrets in both environments. The failure is specifically when the test pod (in obaas-1) tries to verify by connecting to Atlas directly. Additional factors that could differ: Istio Sidecar resource scope — In the test_istio_validation.py , radar applies an Istio Sidecar resource. In escln157, the Sidecar config for the radar namespace may include MongoDB Atlas hosts. In clouddev, the Sidecar for obaas-1 may not. NetworkPolicy — obaas-1 may have stricter K8s NetworkPolicies that block egress to external IPs. The radar-testing env-config — Looking at radar-testing.md , the TENANT_NAMESPACE variable determines everything. Whoever configured clouddev chose obaas-1, while escln157 used a dedicated radar namespace with appropriate network rules.     Q1: Is the radar image built from the radar project and deployed as a pod to run test cases? Yes, your assumption is correct. Here's exactly how it works: Building the image: The .gitlab-ci.yml uses the devops-toolkit docker-images pipeline to build a Docker image:

    radar/.gitlab-ci.yml¶

    variables: CONTAINER_IMAGE_NAME: infra/radar The Dockerfile is simple: FROM $SBCP_CONTAINER_REGISTRY/sbcp/cicd-tools:base-$SBCP_CICD_TOOLS_VERSION COPY ./ /root/ # Copies ALL radar code (tests, libs, config, expectations) WORKDIR /root RUN pip3 install -r requirements.txt CMD ["python3", "-m", "pytest"] # Default command is pytest So the built image (461232396433.dkr.ecr.eu-west-1.amazonaws.com/infra/radar:) contains the entire radar codebase including all tests, libraries, expectation files, and configuration. Deploying as a pod: When the radar-pipeline's deploy-internal-test-job stage runs, it does: helm install sbcp-radar infra/sbcp-radar --namespace=$TENANT_NAMESPACE --set testTarget="internal" ... This Helm chart creates a Kubernetes Job that runs a pod using that radar image. Inside the pod, the command is: pytest -m "internal" tests --junitxml="internal-test.xml" How the specific test cases are triggered: The --set testTarget="internal" becomes the pytest marker expression. So only tests decorated with @internal run inside the pod. If you want specific sub-tests, you can pass additional markers via RADAR_TARGET variable (e.g., "internal and mongo" would only run internal mongo tests). Environment variables are injected into the pod via: configMapRef: environment-config (contains MONGODB_HOST, MONGODB_PROTOCOL, etc.) configMapRef: mongo-config (optional mongo-specific config) secretRef: mongo-cred (optional mongo credentials) configMapRef: cicd-vars- (CI/CD variables from the pipeline) env: TENANT_NAMESPACE (set to the Helm release namespace) Q2: There are three types of tests (internal, external, infra). Why does only internal come with the setup? Correction first: There are three test types: internal, external, and infra (infrastructure). The setup serves both internal AND external, not just internal. Look at the setup stage marker expression:

    From radar-pipeline/main.yml¶

    SETUP_TARGET: "internal or external and not infra" pytest -m "$setup_final_target" setups This means: "run setup functions that are marked @internal OR @external, but NOT @infra." Why infra doesn't need setup: Infra tests validate existing AWS/EKS resources (cluster version, instance types, backup policies, security configs). These resources already exist because the environment was already provisioned. Infra tests only READ and verify — they don't need anything created beforehand. Internal tests need setup because they test the operator's BEHAVIOR. To test "does the mongo operator create a database correctly?", you first need to give the operator a CR to work with. Without the CR, there's nothing for the operator to do, and nothing for the test to verify. External tests sometimes also need setup — for example, creating an Ingress resource (httpbin) so external tests can later hit that endpoint and verify routing works. In summary: Test Type Needs Setup? Why? Infra No Tests existing infrastructure (AWS, EKS). Already provisioned. Internal Yes Tests operator behavior. Must create CRs first so operator reacts. External Sometimes Tests external endpoints. May need services/ingresses deployed first. Q3: What do you mean by "internal test as Kubernetes Job"? Is it a CronJob or a normal Deployment? It is a Kubernetes Job — not a CronJob, not a Deployment. From the sbcp-radar Helm chart template: apiVersion: batch/v1 kind: Job metadata: name: radar-{{ $.Values.contextName }}-{{ $key }} namespace: {{ key }} spec: template: spec: containers: - name: main image: "{{.Values.sbcpImageRepository }}/{{ .Values.imageName }}:{{.Values.imageTag }}" command: [ "/bin/bash" ] args: - -c - >- pytest -m "internal" tests --junitxml="internal-test.xml" ... restartPolicy: Never backoffLimit: 0 The three Kubernetes workload types explained: Type Behavior Use Case Deployment Keeps pods running forever. If a pod dies, it creates a new one. Web servers, operators, always-on services CronJob Creates a Job on a schedule (like Linux cron). Scheduled backups, nightly reports Job Creates a pod, runs it ONCE to completion, then stops. One-time tasks, batch processing, TEST EXECUTION Why a Job is perfect for radar: It runs pytest once, produces a report, and exits backoffLimit: 0 means "if the pod fails, don't retry" (we want to see the actual failure) restartPolicy: Never means "if the container exits, don't restart it" After the tests complete, the pod stays in Completed state so the pipeline can read its logs/output The lifecycle is: helm install → Job created → Pod starts → pytest runs → tests complete → Pod status: Completed → Pipeline reads results → helm uninstall → Job deleted Q4: Every test function is under internal, external, or infra. Q5: What is the concept of namespace in radar testing? Can I pass any namespace? What's the difference between obaas-1, obaas-2, and radar? The namespace concept: TENANT_NAMESPACE is the Kubernetes namespace where: The radar Job pod is deployed The setup CRs (MongoAtlas, MongoAtlasUser, etc.) are created The operator creates Secrets (with credentials) The test pod reads those Secrets and attempts connections Can you pass any namespace? Technically yes, but NOT any arbitrary namespace will work. The namespace must: Already exist in the cluster Have the required RBAC (ServiceAccount with permissions to read secrets, configmaps) Have the necessary network access (Istio egress rules if mesh is enabled) Have the environment-config ConfigMap (contains MongoDB host, protocol, etc.) Radar does NOT create the namespace. It expects it to already exist. The namespace is created during environment provisioning by the sbcp-env-pipeline. Difference between namespace types: Namespace What it is Who creates it Istio? Q12 (Link to Q7): What is the namespace mutex? And explain the Q7 summary again. Your Question: What is the namespace mutex? And I'm not getting your summary of Q7. Answer: Namespace mutex: A mutex (mutual exclusion lock) is a programming concept that prevents two operations from happening at the same time. The namespace mutex in the mongo-operator prevents concurrent reconciliations from corrupting each other within the same namespace. Problem it solves: Imagine two MongoAtlasUser CRs are created simultaneously in namespace obaas-1: CR-1: "create user for app-a with databases [obaas-1_app-a]" CR-2: "create user for app-b with databases [obaas-1_app-b]" Both reconcile functions run at the same time. Both try to update the tenant user's database list. Without a mutex: Thread 1 reads: tenant user has databases [obaas-1_existing] Thread 2 reads: tenant user has databases [obaas-1_existing] Thread 1 updates to: [obaas-1_existing, obaas-1_app-a] Thread 2 updates to: [obaas-1_existing, obaas-1_app-b] ← OVERWRITES Thread 1's change! With namespace mutex: Thread 1 acquires lock for "obaas-1", proceeds Thread 2 tries to acquire lock for "obaas-1", WAITS Thread 1 finishes, releases lock Thread 2 acquires lock, reads UPDATED list, proceeds correctly The code: releaseNamespaceMutex := utils.AcquireNamespaceMutex(reqLogger, cr.Namespace) defer releaseNamespaceMutex() // ... all operations here are protected from concurrent access within same namespace Two CRs in DIFFERENT namespaces can reconcile in parallel (different mutex per namespace).
